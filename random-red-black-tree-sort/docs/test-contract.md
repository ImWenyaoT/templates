# Test Contract: Red-black Tree Sort

## 项目目标

这个项目实现一棵红黑树，用随机顺序插入大量数据，最后输出有序结果。

更通用地说，它练的是：

> 即使输入顺序很乱，系统内部也要一直保持稳定规则，最后给出正确结果。

OpenAI 类比：

- 插入数据像不断接收外部事件。
- 树的平衡规则像 **guardrails**。
- `validate` 像每轮执行后的自检。
- 10,000 条随机数据测试像一个压力更大的 **eval**。

## 输入

输入是：

- 一组值，比如数字或对象。
- 一个 comparator，也就是排序规则。
- 随机插入顺序。

`comparator` 这个词如果要解释给非技术同学，可以说：

> comparator 就是“谁排前面”的比较规则。

## 整理和检查

项目本身不负责校验业务输入，而是假设 caller 提供合理 comparator。

测试里会把输入整理成几类：

- 空数组。
- 单个值。
- 已排序输入。
- 逆序输入。
- 重复值。
- 大量随机值。
- 对象和字符串。

## 核心处理

核心逻辑是：

- 插入节点。
- 必要时旋转。
- 必要时改颜色。
- 保持红黑树规则。
- 用中序遍历输出有序数组。

这里的“旋转”和“改颜色”不用讲太玄。面试里可以说：

> 树在插入后会自动调整形状，避免退化成一条链。

## 输出

输出包括：

- `toArray()`：有序结果。
- `size()`：插入数量。
- `height()`：树高度。
- `metrics()`：插入、旋转、改颜色次数。
- `validate()`：规则检查结果。

## 不能破的规则

这些规则是本项目最重要的 **invariants**。  
人话解释：invariant 就是“无论怎么运行都必须一直成立的规则”。

- 输出数组必须按 comparator 排序。
- 插入多少个值，size 就是多少。
- 空树必须有效。
- 根节点必须是黑色。
- 红色节点下面不能直接接红色节点。
- 从任意节点到叶子的黑色节点数量要一致。
- 大量插入后高度不能退化得太离谱。

## 测试证明

当前测试证明了：

- 空树、单节点、多节点都有效。
- 普通数字、负数、重复值、大数都能排序。
- 对象和字符串可以用自定义 comparator 排序。
- 左旋、右旋、左右旋、右左旋都被覆盖。
- 顺序插入、逆序插入、混合插入后规则仍成立。
- 每次插入后都能立刻 validate。
- 10,000 个随机顺序插入后仍然有序且规则有效。
- 随机工具不会修改原数组，并可用 seed 复现。
- benchmark 工具能返回耗时，也会把执行错误抛给 caller。

## 面试讲法

可以这样讲：

> 这个项目的重点不是“我会写红黑树”，而是我用红黑树练了一种更通用的可靠性方法：复杂状态每变化一次，就检查系统规则有没有被破坏。用 OpenAI 的说法，这有点像每次 tool 执行后都跑一组 guardrails/evals，保证系统没有走到非法状态。
