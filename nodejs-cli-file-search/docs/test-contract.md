# Test Contract: Node.js CLI File Search

## 项目目标

这个项目做一件小但完整的事：用户在命令行输入搜索词和目录，程序返回匹配的文件路径。

更通用地说，它练的是：

> 把命令行输入整理成明确参数，再调用一个可靠的文件搜索工具。

OpenAI 类比：

- `searchFiles` 像一个 **tool**，也就是“可被调用的能力”。
- CLI 参数像 **tool call input**，也就是调用工具时传进去的结构化参数。
- 测试像 **evals**，用固定样例证明工具行为稳定。

## 输入

用户可以输入：

- `search <keyword>`：要搜索的关键词。
- `--root <directory>` 或 `-r <directory>`：从哪个目录开始找。
- `--max <number>` 或 `-m <number>`：最多返回多少条。

这些输入都来自外部，所以不能直接相信。

## 整理和检查

程序需要先做几件事：

- 判断 command 是否是 `search`。
- 判断 keyword 是否存在。
- 把 `root` 解析成真实路径。
- 把 `max` 转成数字，并确认它是正整数。
- 对搜索词做 trim 和小写处理。

这里的 OpenAI 常用词是 **schema**。  
人话解释：schema 就是“输入检查规则”，规定什么输入算合格。

## 核心处理

核心逻辑是 `searchFiles`：

- 从 root 目录开始递归找文件。
- 只匹配文件名，不读取文件内容。
- 匹配时忽略大小写。
- 默认跳过 `.git`、`node_modules`、`dist`、`coverage`。
- 返回绝对路径。
- 按稳定顺序返回，避免每次结果顺序乱变。

## 输出

成功时：

- 每一行输出一个匹配文件路径。
- 没有结果时返回成功，但不输出内容。

失败时：

- 返回非 0 exit code。
- stderr 给出原因。

## 不能破的规则

这些规则类似 **guardrails**。  
人话解释：guardrails 就是保护栏，防止系统越界。

- 空搜索词不能乱搜。
- `--max` 不是正整数时必须失败。
- 默认不能进入依赖目录和构建目录。
- 返回结果必须是文件路径，不是目录路径。
- 结果顺序必须稳定。
- 搜索根目录不存在时要报错，不能假装成功。

## 测试证明

当前测试证明了：

- CLI help、未知 command、缺少 keyword 都有明确行为。
- `--root`、`--max`、短参数、equals 写法都能解析。
- 文件名匹配是大小写不敏感的。
- 空白搜索词返回空结果。
- 搜索结果是绝对路径。
- 默认 ignored dirs 会被跳过。
- caller 可以替换 ignored dirs。
- `maxResults` 会限制输出数量。
- 文件内容不会被误当成文件名匹配。
- 缺失目录和非法参数会返回错误。

## 面试讲法

可以这样讲：

> 这个 CLI 项目不是只写一个搜索脚本。我把它拆成两层：命令行负责把用户输入整理成可靠参数，`searchFiles` 负责真正搜索。测试覆盖了正常搜索、坏参数、目录边界、默认忽略目录和稳定排序。用 OpenAI 的说法，它像一个小 tool：输入有 schema，输出有固定格式，evals 用来证明工具行为稳定。
