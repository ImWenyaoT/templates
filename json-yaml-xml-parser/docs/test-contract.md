# Test Contract: JSON/YAML/XML Parser

## 项目目标

这个项目接收 JSON、YAML、XML 三种格式的数据，把它们统一转成标准 JSON 可以表达的结果。

更通用地说，它练的是：

> 把不同格式的外部内容，整理成一个统一、可验证的内部数据形态。

OpenAI 类比：

- 格式检测和解析像输入整理。
- 标准 JSON 输出像 **Structured Outputs**。
- 错误处理像 **guardrails**。
- 测试样例像 **evals**。

## 输入

输入可以来自：

- 文件路径，比如 `input.json`、`input.yaml`、`input.xml`。
- 显式格式参数，比如 `--format json`。
- 文件内容。

这些输入都可能出错：

- 文件可能不存在。
- 扩展名可能不支持。
- 内容可能不是合法 JSON/YAML/XML。
- 解析结果可能无法安全转成 JSON。

## 整理和检查

程序先做这些事：

- 如果传了显式格式，就优先使用显式格式。
- 如果没有显式格式，就从文件扩展名判断格式。
- 只接受 `json`、`yaml`、`xml`。
- 拒绝未知格式和无扩展路径。
- 读取文件后交给对应 parser。

这里的 **schema** 不只是对象字段检查，也可以理解成“格式规则”。  
人话解释：先规定输入必须是哪几类，再继续处理。

## 核心处理

核心逻辑是：

- `detectFormat`：判断该用哪个 parser。
- `parseDocument`：解析内容并返回数据和元信息。
- `parseData`：只返回解析后的数据。
- `toSerializableValue`：确保输出能被 JSON 表达。

XML 会被整理成稳定对象结构：

- 属性放进 `$attributes`。
- 文本放进 `$text`。
- 重复子元素变成数组。

## 输出

成功时：

- 返回标准 JSON 可表达的数据。
- CLI 输出 pretty JSON。

失败时：

- 返回明确错误，比如文件不存在、格式不支持、解析失败。

## 不能破的规则

- 不支持的格式必须拒绝。
- 没有路径也没有显式格式时必须拒绝。
- 输出必须能被 JSON 表达。
- `undefined` 对象字段要被安全处理。
- `NaN`、`Infinity`、`BigInt`、`Symbol` 不能偷偷进入输出。
- XML 的属性、文本、重复节点不能丢。

这些就是项目的 **guardrails**：系统不能越过的底线。

## 测试证明

当前测试证明了：

- `.json`、`.yaml`、`.yml`、`.xml` 能自动识别。
- 显式 `--format` 和短 `-f` 能覆盖扩展名。
- 无扩展路径和未知扩展会失败。
- JSON/YAML/XML 的对象、数组、标量、嵌套结构能解析。
- XML 属性、文本、重复节点能保留。
- CLI 可以解析相对路径和绝对路径。
- 文件不存在、目录路径、坏格式都会返回清楚错误。
- 输出会拒绝不可 JSON 表达的值，并指出嵌套路径。

## 面试讲法

可以这样讲：

> 这个 parser 项目练的是把多种外部格式统一成一个可靠输出。它先判断输入格式，再用对应 parser 处理，最后用 `toSerializableValue` 保证输出一定能被 JSON 表达。用 OpenAI 的说法，它接近 Structured Outputs 的思路：不只把东西解析出来，还要保证最后结果长成调用方能稳定消费的格式。
